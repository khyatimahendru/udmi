# Key Rotation

Key rotation in UDMI allows devices and the cloud to securely cycle the authentication keys used for MQTT connections. Key rotation can be initiated either by the device or by the cloud.

## 1. Device-Generated Key Rotation

In this scenario, the device generates a new public/private key pair.

1. **Generation:** The device generates its own key pair locally.
2. **Publishing:** The device publishes its new **public key** as part of a system event (`events/system`). The event payload includes a `credentials` array that matches the schema defined in `model_cloud.json`.
3. **Provisioning:** UDMIS (specifically the `TargetProcessor`) intercepts the system event. When it detects a populated `credentials` array, it provisions the new public key to the cloud IoT provider using the `iotAccess.modelDevice()` interface.
4. **Reconnection:** The device disconnects from the broker and reconnects using its newly generated private key.

## 2. Cloud-Generated Key Rotation

In this scenario, the key pair is generated by the cloud and securely delivered to the device.

1. **Generation:** The cloud IoT infrastructure generates a new key pair.
2. **Delivery:** The cloud delivers the **private key** to the device as a system blob configuration. The blob key used for this is `_iot_endpoint_secret`, which is a predefined system blobset (similar to `_iot_endpoint_config`).
3. **Application:** The device receives the config update, applies the new private key, and securely stores it.
4. **Reconnection:** The device disconnects from the broker and reconnects using the new private key.

## Schemas

* **`schema/common.json`**: The `_iot_endpoint_secret` blobset key is defined in the `blobsets` enum to facilitate cloud-generated key delivery.
* **`schema/events_system.json`**: The `credentials` array is defined to facilitate device-generated key publishing. It supports formats such as `RS256`, `ES256`, and their X509 variants.
